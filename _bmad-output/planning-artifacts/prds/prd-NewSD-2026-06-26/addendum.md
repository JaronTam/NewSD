## 1. 技术选型论证（自 Idea 文件，PRD 第七节为其概要）

### 1.1 渲染层

- 事实：DOM 节点超过 80×60 网格高频刷新引发 Layout 抖动
- 推导 → 必须用 VRAM 双缓冲区（文本缓冲 + 颜色缓冲）
- 边界约束 → 霓虹字体 shadowBlur 极端消耗 GPU
- 结论：Canvas 2D 定点渲染二维字符数组，关闭抗锯齿保证像素风

### 1.2 模拟引擎

- 事实：用户在面板配置复杂流量公式
- 推导 → 前端必须动态解析代数环
- 边界约束 → 防止 JavaScript 注入 + 提升高频迭代性能
- 结论：Rust 手写递归下降 parser 构建 AST（mexpr/meval 不存在于 crates.io，已弃用）+ autodiff 0.7.0 自动微分 + faer 0.24.4 LU 分解；Wasm 层拓扑排序，检测到非法环立即断流抛出（详见 ARCHITECTURE-SPINE.md AD-6）

### 1.3 后端

- 事实：ASCII 结构天然高压缩率
- 推导 → 画板数据本质是纯文本，无需复杂解析
- 边界约束 → 轻量化、极速读写，且 MVP 为单节点、10 人匿名协作
- 结论：服务端 SQLite 存储画板与 CRDT 快照，Presence 与仿真态走服务进程内存

## 2. 七大数学约束的工程机制（PRD 第二节为产品级表述）

PRD 第二节将这些约束表述为建模原则/产品行为；本节记录其工程实现机制，供架构阶段使用。

| 约束 | 工程机制 |
|------|----------|
| 存量断路器 | 拓扑排序前强行剪断所有存量流出边，区分"积分反馈环"与"代数死锁环" |
| 单次求值 | 所有流量速率每步只算一次，查表供所有存量使用，消除浮点漂移导致的物质凭空产生/消失 |
| 非负钳制 | 存量<0 时限制流出，级联影响下游；隐式法下为带约束非线性求解（投影/活动集），见 §7 |
| DELAY 隐式存量 | 延迟函数不做历史队列，编译期转为串联幽灵存量，保证隐式求解器联立求解时的连续性 |
| 双时间轴分离 | UI 时间轴 ≠ 仿真时间轴，防止刚性系统数值爆炸传导到画布 |
| NaN 熔断 | Wasm 每帧输出后做位运算检查，正反馈上溢时紧急停机 |
| 两遍扫描初始化 | Pass0 算初始值 → Pass1 焊死后推导流速，解决 t=0 时序死锁 |

> ⚠️ 已知风险（待架构阶段规约）：非负钳制与隐式牛顿迭代叠加构成带约束非线性求解（投影法或活动集法），实现非平凡——钳制会改变活动约束的方程结构与雅可比，须专门规约收敛性与级联重算策略。PRD FR-SIM-4 已标注此点。详见 §7。

## 3. 持久化与扩展路线图（C1 决策落地）

### 3.1 MVP 存储（决策）

- 服务端 SQLite：CRDT 快照 + 追加式操作日志（单文件、服务进程内访问）
- 进程内存：Presence（光标/在线）+ 仿真态快照（FR-COLLAB-5）
- 单节点部署，匹配 10 人匿名协作规模

### 3.2 后续路线图（被排除出 MVP）

| 组件 | 触发条件（可观测阈值，待架构实测校准） | 迁移动机 |
|------|----------|----------|
| PostgreSQL | SQLite 写延迟持续 >200ms（滑动窗口 p95），或日活 >200 用户，或单库 >10GB | SQLite 单写者瓶颈、跨节点共享 |
| Redis | 部署 ≥2 个服务实例（Presence 与仿真态需跨实例共享时） | 单进程内存无法跨实例 |
| 多节点 WebSocket 网关 | 单节点 WebSocket 并发连接 >500（或单节点 CPU 持续 >70%） | 水平扩展协作中继 |

> 阈值说明：上述阈值为架构阶段前的估计值，须在 MVP 上线后以实测负载校准（"先垂直后水平"原则：先榨干单节点——垂直扩容 + SQLite WAL + 连接复用，再触发水平迁移）。触发条件均为可客观观测指标，避免"规模增长""需要"等主观判定导致迁移决策无据可依。

### 3.3 被拒绝的替代方案

- **嵌入式 SQLite（无服务端）**：被否。实时多人协作必须有服务端做 CRDT 中继，"嵌入式"与协作模型根本冲突。
- **MVP 即上 PostgreSQL+Redis**：被否。10 人匿名单节点场景下引入运维负担与部署复杂度，收益为零，违反 YAGNI。

## 4. 仿真结果数据通路（C4/C5 前置依赖，待架构统一）

MVP 与后续能力对仿真结果数据的消费方式不同，架构阶段须统一数据通路：

| 消费方 | MVP? | 所需数据形态 |
|--------|------|--------------|
| sparkline（FR-SIM-VIZ-1） | ✅ MVP | 单存量近 N 步采样点 |
| 完整时序面板 L3 | ❌ 后续 | 多存量全时序曲线 |
| 挑战关卡达标判定（FR-GAME-3） | ❌ 后续 | 用户仿真曲线 vs 目标阈值/区间的数值比对 |

**架构要点（开放问题 4）**：达标判定引擎与 L3 时序面板可能复用同一"仿真结果时序缓冲"数据通道。二者均为后续能力，但若 MVP 的 sparkline 采样缓冲设计得当，可为二者复用铺路。需在架构阶段定数据通路，避免重复造轮子。

## 5. CLD 后续能力说明（C3 决策）

MVP 仅 SFD。CLD（因果回路图）作为后续能力，其与 SFD 的关系需在引入时设计：

- CLD 图元：变量节点 + 因果链接（带 +/- 极性），定性、无数值
- SFD 图元：存量/流量/源汇，定量、有方程
- 待定：两图混排方式、CLD→SFD 是否提供自动/半自动转换
- 已删除项（不回归）：flow 的 `polarity` 字段、`⊕/⊖` 反馈标识图元——这些是 CLD 概念误套到 SFD 的范畴错误，CLD 引入时应作为 CLD 专属图元，而非回填到 flow

## 7. 求解器与数值子系统（③ 决策落地）

### 7.1 决策

MVP 采用隐式求解器（后向欧拉 / BDF），每步牛顿迭代收敛，适用于刚性系统。用户在讨论中明确选择此方案并确认全进 MVP，覆盖了推荐方案（固定 RK4 / 自适应 RK45）。

### 7.2 工程分量（架构阶段须专项规约）

隐式求解器引入完整的数值线性代数子系统：

- **自动微分**：雅可比由公式 AST 自动微分得到，与 FR-SIM-7 量纲 AST 复用——这使量纲 AST 不只做量纲校验，还须支撑雅可比生成
- **牛顿迭代**：每步非线性方程求解，含收敛判定（残差/步长阈值）与失败回退
- **线性求解器**：LU 分解求解牛顿步的线性方程组
- **雅可比近似/更新策略**：是否每步重算、是否用 Broyden 近似

### 7.3 体量警示（如实记录）

隐式求解器 + 量纲引擎（①）+ 自动微分 + 牛顿迭代 + 线性求解器，使 MVP 数值子系统接近专业数值软件复杂度，与"轻量匿名游戏化探索沙盒"的产品定位存在张力。本警示已向用户提示，用户仍确认全进 MVP。后续若 MVP 范围需回调，求解器降级（隐式 → 自适应 RK45 → 固定 RK4）是最有效的减负杠杆。

### 7.4 与断路器的交互（决策）

隐式法下 FR-SIM-2 仍保持严格剪边：代数环一律拒绝，不因隐式可解而放行。理由：SD 中代数环通常为建模错误，严格剪边利于在探索中直观区分"合法积分反馈"与"代数死锁"。

### 7.5 被拒绝的替代方案

- **固定步长 RK4 + 弱化刚性承诺**：被否（用户选隐式）。
- **自适应 RK45**：被否（用户选隐式）。

## 8. 公式语义感知合并（⑤ 决策落地）

### 8.1 决策

MVP 采用方案 C 语义感知合并：流量公式以结构化 AST 存于 CRDT，节点级合并 + AST↔文本双向同步编辑器。用户在讨论中明确选择此方案并确认全进 MVP，覆盖了推荐方案 A（检测+提示）。

### 8.2 工程分量

- **结构化公式 CRDT**：FR-COLLAB-1 文档模型改造，公式从扁平字符串改为 `Y.Map` 嵌套 AST
- **AST↔文本双向同步**：编辑器实时解析文本→AST 写 CRDT，CRDT→文本序列化显示
- **不完整语法处理**：输入中途（如 `人口*`）以"待定节点"承载，不阻断输入、不丢光标位置
- 与 ② id 引用利好耦合：AST 引用节点存 stockId，节点稳定，利于结构化合并
- 与 ① 量纲校验利好耦合：合并后 AST 始终可解析，量纲校验前置稳定

### 8.3 体量警示（如实记录）

结构化公式 CRDT + AST↔文本同步编辑器属协作编辑器领域硬骨头（类比代码协作编辑器的 AST 同步）。叠加 §7 隐式求解器数值子系统，MVP 已是"协作版专业 SD 数值平台"体量，与"轻量匿名游戏化探索沙盒"定位存在显著张力。本警示已第四次向用户提示，用户仍确认全进 MVP。

后续若 MVP 范围需回调，减负杠杆优先级：⑤ 语义合并降级（C→A 检测提示）> ③ 求解器降级（隐式→自适应→固定）> ① 量纲降级（C→B 显示用 units）。

### 8.4 被拒绝的替代方案

- **方案 A 检测+提示**：被否（用户选 C）。
- **方案 B 字段级锁**：被否（协作感差）。

#### #2 非负钳制雅可比更新策略未覆盖 — `FR-SIM-4`(R1 high)

- **架构收口**:
  1. 约束激活后雅可比是否立即重算(策略:全重算 / Broyden 近似 / 局部修正)
  2. 防 Chattering 机制:hysteresis band(滞回带,激活与释放阈值分离)
  3. 投影法 vs 活动集法优先级选择(MVP 建议:活动集法优先,投影法兜底)
- **关联**: addendum §2 已如实标注此难点;addendum §7(求解器专项)须展开约束求解收敛性与雅可比适配。

### 协作专项(4 条,建议合并为「协作冲突与迁移规约」一节收口)

#### #3 AST 结构性冲突未讨论 — `FR-COLLAB-6`(R2 high)
- **出处**: review-collab §1.2;PRD FR-COLLAB-6 称"结构化合并保证 AST 始终可解析"过乐观。
- **问题**: 三类结构性冲突——括号插入改变结合性(场景A)、运算符替换(场景B)、删除子树产生幽灵编辑(场景C)。合并结果可能是合法 AST 但语义错误。
- **架构收口**:
  1. AST→Y.Map 显式映射 schema(每节点类型字段布局,杜绝序列化为 Y.Text/Y.XmlFragment 等非预期类型)
  2. 结构性冲突检测(合并后拓扑比较预期树结构)
  3. 冲突回退行为:锁定相关子树 / 标记冲突区域 / 回退文本级(三选一或分级)
- **关联**: 与 #4 待定节点、#6 降级接口耦合。

#### #4 待定节点 CRDT 表示/合并未定义 — `FR-COLLAB-6`(R2 high)
- **出处**: review-collab §2.3;PRD FR-COLLAB-6 仅说"待定节点承载不完整语法"。
- **问题**: 待定节点在 Y.Map 中数据结构、合并规则、待定→完整转换触发条件均未定义,合并行为不可预测(输入丢失 / AST 状态不一致 / 光标错位)。
- **架构收口**:
  1. 待定节点 Y.Map schema(建议 `type: "pending"` + `rawText` 残片)
  2. 合并规则(Y.Text 插入合并 vs Y.Map 键级覆盖,选一种)
  3. 待定→完整转换触发条件(解析器无剩余语法错误 / 用户确认)
  4. 附带 medium(同源):待定节点对量纲校验的处理策略(review-collab §2.1,建议输入中途只做语法高亮、AST 完整后才触发量纲校验)
  5. 附带 medium:待定节点跨流量/跨画板复制行为(review-collab §2.2,粘贴含待定节点公式须校验可解析)
- **关联**: #3 schema 定义后,待定节点 schema 作为其子集。

#### #5 快照与 CRDT 文档时序对齐缺口 — `FR-COLLAB-3`/`FR-COLLAB-5`(R2 high)
- **出处**: review-collab §3.1;PRD FR-COLLAB-5(仿真态快照,进程内存)+ FR-COLLAB-3(新房主续跑)。
- **问题**: 快照采集时刻与 CRDT 同步时刻可能错位——新房主用旧模型结构的仿真结果 + 新初始值续跑,可能跳跃/不连续,**违反"无 t=0 回滚"承诺**(反指标)。
- **架构收口**:
  1. 快照附 CRDT 文档版本标识(Y.Doc `clientID` + `clock` 向量)
  2. 新房主拉取快照时校验对齐:对齐→直接续跑;不对齐→增量重跑(从快照时刻到当前 CRDT 版本的仿真追赶)
  3. 不采用"阻塞 CRDT 变更直到快照采集完成"(阻塞协作不可取)
- **关联**: 直接关系反指标"房主迁移导致仿真状态清零 = 0",架构须保证。

#### #6 ⑤ 减负杠杆 C→A 回滚成本被低估 — addendum §8.3(R2 high)
- **出处**: review-collab §6;addendum §8.3 列减负优先级 ⑤>③>①。
- **问题**: ⑤ 降级(结构化 AST→扁平字符串)非"改类型",涉及 CRDT 文档模型迁移、AST 感知编辑器重写、协作冲突退化为文本级,成本曲线非线性。
- **架构收口**: 显式定义「降级接口」(⑤→A 适配器层),预留三项前提(否则降级时需重构多模块):
  1. 数据兼容层:CRDT 快照读取兼容两种 formula 格式(Y.Map vs string)
  2. 编辑器抽象:AST 感知编辑器可替换为普通文本编辑器
  3. 校验管道抽象:量纲校验与 autodiff 不硬依赖 CRDT AST,接受"任意来源 AST"
- **关联**: 这是 MVP 体量逃生阀(§5.1)可否真实触发的工程前提;若三项未预留,⑤ 降级名存实亡。

### 安全专项(1 条)

#### #7 Wasm 求解器缺时间维/迭代次数维熔断 — `FR-SIM-6`(R3 high)
- **出处**: review-security §5;PRD FR-SIM-6 仅值域熔断(NaN/Inf/量级突变)。
- **问题**: 恶意/病态公式可致牛顿迭代无限不收敛(不产生 NaN)、单步 wall-clock 无限延长。无 maxIterations、无 AST 复杂度预算、无 Wasm 内存硬限制。
- **架构收口**(FR-SIM-6 扩展,与值域熔断同级保护):
  1. 单步 wall-clock 超时(建议每步 ≤500ms),超时熔断报 `SOLVER TIMEOUT`
  2. 牛顿迭代 maxIterations(建议 100),仍不收敛则熔断
  3. AST 编译阶段复杂度预算:拒绝超过 MAX_NODES(建议 5000 节点)的公式
  4. Wasm 实例内存上限:WebAssembly.Memory 上限(建议 64MB),超出重建实例
  5. 所有熔断以 ASCII 通知反馈(类比 FR-SIM-6 `[SYSTEM HALTED: ...]`)
- **关联**: 与 #1 牛顿收敛性、FR-SIM-8 降级链边界须区分——#7 是资源耗尽熔断(暂停报错),FR-SIM-8 是收敛性压力(降级不阻断)。

**架构阶段须补的运行时细则**:

1. **收敛监控阈值**(FR-SIM-8 默认策略依赖):
   - 残差范数超阈判定(建议 ‖F(y)‖₂ < tol,tol 与 dt、模型量级关联,架构定具体值)
   - 步长收缩率判定(建议连续 N 步步长缩减因子 < ρ 视为步长崩溃,N/ρ 架构定)
   - 与 #1 步长控制策略、#2 雅可比更新策略联动——收敛监控的阈值须在求解器专项规约中一并标定

2. **降级触发阈值**:
   - BDF→后向欧拉的切换条件(残差超阈次数 / 步长崩溃)
   - 后向欧拉→显式试探步的切换条件(后向欧拉仍不收敛次数)
   - 显式试探步仍失败→触发二级单按钮(预设降级组合)

3. **预设降级组合**(二级"简化模型以继续"自动执行的内容,PRD 仅说"减少存量/降低耦合"):
   - 具体降级操作集(如:移除 DELAY 链末端 / 合并高耦合存量 / 降低 DELAY 阶数 / 切换无量纲校验)
   - 降级优先级(对模型洞察力损伤由小到大,与 §5.1 逃生阀 ⑤>③>① 口径一致)
   - 降级后须向用户透明告知"模型已被简化"的具体项(状态栏清单,非弹窗)

4. **与既有机制边界**(PRD 已定,架构落实):
   - FR-SIM-8 = 收敛性压力,不阻断(降级/单按钮续跑)
   - FR-SIM-6(+ #7 扩展)= 数值域/资源耗出,熔断暂停报错
   - §5.1 逃生阀 = MVP 体量超载,架构阶段触发,非运行时
   - 三者层次不同,架构须保证运行时不混淆触发

### #8 渲染裁决:per-glyph shadowBlur vs VRAM 双缓冲(open,可暂停交付)

- **出处**:PRD §2.1 将"霓虹字体 shadowBlur 极端消耗 GPU"列为边界约束,结论为"Canvas 2D 定点渲染 + 关闭抗锯齿 + VRAM 双缓冲";lovable/prototype 现实现采用全帧重绘 + per-glyph shadowBlur(glowFor/textShadow),视觉质感即建立在此被 PRD 拒绝的技术上。
- **问题**:prototype 未验证 PRD 的 VRAM 方案能呈现同等霓虹质感——它验证的是会被 1000 图元 @ 60FPS(PRD §1.4/§4.1 指标)否决的方案。质感与性能承诺存在未决张力,既有的 7 条 open high 未覆盖(均在求解器/协作/安全专项)。
- **架构收口**(二选一裁决):
  1. **(a) VRAM 路径**:辉光图集(glow atlas)+ VRAM 字符缓冲(文本/颜色双区)+ 色相偏移 Shader 复刻霓虹质感,禁用 per-glyph shadowBlur。FR-UI-6 第 9、10 项(CRT 背景漂移、per-glyph 辉光)以本方案复刻。
  2. **(b) shadowBlur 路径**:保留 per-glyph shadowBlur,下调 PRD §1.4 与 §4.1 的"1000 图元 @ 60 FPS"指标。**注意:选 (b) 属 PRD 改动,须同步修订 §1.4/§4.1 指标,走 PR**;FR-UI-6 第 9、10 项直接沿用 prototype 实现。
- **可暂停**:本项为可暂停交付项。架构阶段若无法定裁决,可暂缓,不阻断其余架构产物(求解器/协作/安全专项)。
- **关联**:FR-UI-6 第 9、10 项的可实现形态完全由本条裁决决定;FR-CANVAS-5 小地图的 10k 图元性能预算与本条耦合。

### #9 公式 tokenizer 语法要求(open)

- **出处**:PRD FR-ELEM-3 要求 `@<uuid>` id 引用语法 + `数值 [单位]` 常数单位标注语法;lovable/prototype 现 tokenizer(`src/lib/sd/formula.ts`)仅支持 `+ - * / ( ) 数字 标识符`,对 `@` 与 `[` 字符直接抛 `Unexpected` 错误。
- **架构收口**:架构阶段公式 tokenizer 必须扩展支持两种语法:
  1. `@<uuid>` —— 存量引用节点(FR-ELEM-3,见 #10)
  2. `数值 [单位]` —— 常数单位标注(FR-SIM-7 量纲推导参与项)
- **关联**:与 #10 引用模型、FR-SIM-7 量纲校验同源,须在同一 AST 规约中收口。

### #10 公式引用模型:存储 id / 显示 name(open)

- **出处**:PRD FR-ELEM-3 要求公式以 stockId 引用(`@<uuid>*0.1`),编辑器显示 name,重命名零断链、协作并发重命名安全;lovable/prototype 现用 name 引用(`人口*0.05`),env 按 `s.name` 构建——可读性佳但重命名断链、并发重命名不安全。
- **判定**:PRD 口径正确(为协作安全),prototype 为单用户阶段捷径。
- **架构收口**:明确分层——
  1. **存储层**:CRDT 公式 AST 的引用节点存 stockId(UUIDv4),与 FR-COLLAB-1 结构化 AST 一致
  2. **编辑/显示层**:编辑器将 `@<uuid>` 解析渲染为对应存量 name,用户可见可读;重命名存量只改 name 不动 id,公式引用零断链
- **关联**:与 #9 tokenizer、FR-COLLAB-6 语义感知合并、FR-SIM-7 量纲精确解析(以 id 解析被引用存量 units)同源。

# R4 教育定位 / 体量张力 — 审查报告

## 整体判定

PRD 的产品定位（教育/轻量）与 MVP 实际含项（隐式 BDF 求解器 + 牛顿迭代 + LU 分解 + 自动微分 + 结构化 AST CRDT 语义合并）之间存在系统性、被多次记录但未被 PRD 正文充分消化的张力。Addendum §7.3/§8.3 诚实记录了警示，但 PRD 正文的定位段（§1.1-§1.3）未据此调整措辞，导致"教育"包装下实质是一套协作版专业 SD 数值平台。定位诚实度不足。

---

## 逐核查点判定

### CP1: 定位段是否诚实反映体量？

**严重度: HIGH**

**引用位置:** prd.md §1.1（愿景/差异化特点）、§1.2（问题陈述）、§1.3（目标用户）vs addendum §7.3/§8.3

**问题:** PRD 正文 §1.1 未出现"轻量"一词，但 §1.2 批评现有工具"界面过于复杂，入门门槛高"，隐含本工具更简单易用的承诺。§1.3 明确将"学生"和"教育工作者"列为主打用户。然而 MVP 含项包含：隐式 BDF 求解器 + 牛顿迭代 + LU 线性求解 + 自动微分 + 结构化 AST CRDT 语义合并 + 量纲校验引擎——这已构成"协作版专业 SD 数值平台"体量（addendum §8.3 原文）。体量警示被诚实记录在 addendum，但 PRD 正文的定位段未做任何反映该张力的措辞调整（如添加"本工具对教育场景需注意的学习曲线"或"数值子系统面向高级用户"的说明）。定位段与 MVP 体量之间存在修辞层面的不诚实。

**修复建议:**
- §1.1 愿景句应坦陈体量："一款赛博朋克风格的系统动力学建模平台，兼顾教育入门与专业数值能力"——而非暗示轻量
- §1.2 问题陈述应区分"工具复杂度门槛"与"本工具在数值子系统层面的固有复杂度"
- §1.3 应增加定位张力声明，说明教育用户可能需绕过隐式求解器概念

---

### CP2: 目标用户(1.3)是否能驾驭隐式求解器/量纲校验？

**严重度: HIGH**

**引用位置:** prd.md §1.3（目标用户）、addendum §7（求解器子系统）

**问题:** PRD 定义的学生用户（"学习反馈回路和存量流量建模"）和教育工作者（"教授系统动力学概念"）——典型教育场景的用户并不需要、也难以理解隐式 BDF 求解器、牛顿迭代收敛判定、LU 分解、自动微分雅可比、量纲校验 AST 等概念。MVP 的数值子系统面向的是需要刚性系统求解的高级 SD 建模者。PRD 未命名这一张力：教育新手与需刚性系统求解的高级用户之间存在显著的能力需求错配，且 PRD 未说明教育用户如何"跳过"或"无视"这些复杂子系统。§1.3 的"系统思考者"和"爱好者"定位过于泛化，未承担高级用户画像的职能。

**修复建议:**
- §1.3 增加用户分层说明：明确教育用户使用"默认配置的简化界面"，高级用户解锁隐式求解器能力
- 或增加"用户旅程"描述，说明教育场景下隐式求解器作为"透明底层"（用户只看到播放/暂停），量纲校验作为"软警告"可忽略

---

### CP3: 体量警示四次记录是否构成诚实 trade-off？

**严重度: MEDIUM**

**引用位置:** addendum §7.3（求解器体量警示）、§8.3（语义合并体量警示）、.memlog.md 条目 15/17/19

**问题:** 四次警示被如实记录在 addendum 和 memlog，且每次标注"用户已确认全进 MVP"。但从产品定位文档角度看，这些警示存在两个结构性问题：(a) 全部存放在 addendum（工程/架构附录），PRD 正文 §5 MVP 范围列表中对每项工程的体量代价无附带说明；(b) 每次以"用户仍确认全进 MVP"收尾，形成模式化的免责声明——降低了警示作为"真正可回调的 trade-off 记录"的说服力。警示从"诚实的风险记录"滑向"例行合规手续"。

**修复建议:**
- 在 PRD §5.1 MVP 包含列表中，对隐式求解器、语义合并、量纲校验三项加注体量警示标记（如 ⚠️），使读者在阅读 MVP 范围时即时感知代价
- addendum §7.3/§8.3 的"用户确认"措辞改为更具体的决策上下文（如"在已知 X 代价前提下选择"），避免例行格式

---

### CP4: 减负杠杆 ⑤>③>① 的优先级是否合理？

**严重度: LOW**

**引用位置:** addendum §8.3（减负杠杆优先级）

**问题:** 优先级 ⑤（语义合并降级）> ③（求解器降级）> ①（量纲降级）整体合理，论证充分。⑤ 对教育价值损失最小——公式编辑的语义合并（CRDT AST 级合并）对典型教育协作场景（学生各自建模型而非并发编辑同一公式）增益有限，降级到冲突检测提示对教学体验影响可控。③ 降级到自适应 RK45 对教育场景的大多数教科书模型（人口、SIR、logistic 等非刚性系统）影响有限，但 addendum §7.3 未说明刚性教学场景（如化学动力学、多时间尺度生态模型）的具体损失——这是一个小的论证缺口。① 量纲降级（C→B 显示用 units）对教育价值损失最大，作为最后防线合理。

**补充建议（非必须）:**
- 对 ③ 降级的损失补充刚性教学场景的具体示例，帮助决策者理解取舍

---

### CP5: 游戏化徽章(FR-GAME-2)与专业数值子系统的定位割裂

**严重度: MEDIUM**

**引用位置:** prd.md §3.8 FR-GAME-2（行为徽章）、addendum §6（游戏化设计）

**问题:** 游戏化徽章系统面向入门级用户——"首个存量"、"首次连接流量"、"首次跑仿真"等触发器面向零基础学习者。然而同一 MVP 内含 BDF 求解器 + 牛顿迭代 + LU 分解 + 自动微分（工程复杂度接近专业数值软件）。一个"需要理解隐式求解器概念的高级建模工具"同时颁发"你放了第一个存量"徽章，存在明显的用户成熟度错配。PRD 和 addendum 均未承认或解释这一割裂。游戏化总开关（FR-GAME-1）虽允许关闭，但无法解决产品定位的二元性——工具同时面向"连存量都不知道是什么"的学习者和"需要刚性系统求解"的高级建模者。这不是不可解的矛盾（分层 UI 可缓解），但 PRD 未命名它。

**修复建议:**
- §3.8 或 §1.3 增加说明：游戏化层面向教育/入门用户，高级数值能力面向专业用户，二者通过 UI 分层/默认配置隔离
- 或调整徽章触发器范围——当使用隐式求解器/自定义雅可比等高级功能时自动关闭入门级徽章

---

### CP6: sparkline(FR-SIM-VIZ-1)作为 MVP 唯一结果呈现——对教育场景的量纲/单位教学是否充分？

**严重度: HIGH**

**引用位置:** prd.md §3.3.1 FR-SIM-VIZ-1（sparkline）、FR-ELEM-1（存量 units 字段）、FR-SIM-7（量纲校验）

**问题:** PRD 投入了大量工程资源实现量纲校验系统（FR-SIM-7：全 AST 量纲推导 + 常数单位标注 + 模型级时间单位），但 MVP 唯一的结果呈现是 ASCII sparkline——它将数值归一化到 8 档字符高度（▁▂▃▄▅▆▇█），完全抹除了绝对数值和单位信息。对于教育场景，"单位教学"是核心目标之一——学生需要看到"人口(人)随时间增长"、"水位(升)在下降"，而非无量纲归一化曲线。Sparkline 无法显示：
- 存量的当前数值和单位
- 同一面板中不同单位存量的对比
- 量纲校验结果的可视化反馈（如预期"人/year" vs 推导"人"的差异如何影响曲线）

FR-ELEM-1 定义存量有 `currentValue` 和 `units`，FR-UI-3 状态栏显示模拟时间，但 PRD 未要求存量方框内同步显示当前数值 + 单位——这是一个可修复的低成本补丁。更大的问题是 sparkline 作为唯一趋势可视化，对量纲教学严重不足。

**修复建议:**
- FR-SIM-VIZ-1 增加要求：存量方框内（或 sparkline 旁）同步显示当前数值和单位（如 `1234 人`），不归一化
- 或在 FR-UI-3 状态栏增加量纲校验结果概要（如"3/5 流量量纲一致，2 条软警告"）
- 考虑在 MVP 范围内增加"数值面板"的极简版本（显示每个存量的当前值+单位+趋势方向箭头），而非等到 L3 完整时序面板

## 严重度汇总表

| 核查点 | 严重度 | 核心问题 |
|--------|--------|----------|
| CP1 定位段体量诚实度 | **HIGH** | PRD 正文以"教育"修辞平滑了专业数值平台事实，addendum 警示未回灌到定位段 |
| CP2 目标用户驾驭能力 | **HIGH** | 教育用户与隐式求解器/量纲校验的用户能力要求错配未被命名 |
| CP3 体量警示 trade-off | **MEDIUM** | 四次警示模式化，"用户确认"沦为例行免责，且仅存于 addendum |
| CP4 减负杠杆优先级 | **LOW** | ⑤>③>① 优先级合理，仅缺③降级对刚性教学场景损失的细化说明 |
| CP5 游戏化徽章定位割裂 | **MEDIUM** | BDF 求解器工具同时发"首个存量"徽章，用户成熟度错配未命名 |
| CP6 sparkline 教育充分性 | **HIGH** | 量纲系统投入大量工程，但 sparkline 归一化抹除了所有单位信息，教育核心目标落空 |

**严重度计数: HIGH × 3 | MEDIUM × 2 | LOW × 1**

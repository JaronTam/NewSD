## 0. 整体判定

**范围定义基本合理,但存在 2 处重大遗漏与 1 处可量化性缺陷,须在进入架构阶段前修复。**

- 5.1/5.2 的 includes/excludes 互补,无重叠;开放问题全部真开放。
- 但体量警示四次累计仅藏于 addendum,PRD 主文档 5.x 未内置逃生阀,这是最大风险。
- 三条后续路线触发条件不可观测/不可量化,架构阶段无法据此做客观决策。
- C3/C4 的"后续"边界清晰,无回流风险。

## 1. 核查点一:5.1 MVP Includes 逐项 Challenge

### 1.1 ✅ ASCII 渲染无限画布
**判定**: MVP 必要。无画布即无产品。零争议。

### 1.2 ✅ 存量/源汇/流量 三种图元 (SFD)
**判定**: MVP 必要。SFD 是产品核心建模语义。零争议。

### 1.3 🔶 隐式求解器 (后向欧拉/BDF + 牛顿迭代) + 存量断路器算法
**严重度**: Major  

**Challenge**: addendum §7.3 自认"MVP 数值子系统接近专业数值软件复杂度,与轻量匿名教育工具定位存在张力"。对教育场景 MVP,固定步长 RK4 即可覆盖常见教育模型(人口增长、库存振荡、捕食者-猎物),工程成本低 1-2 个数量级。

**决策惯性嫌疑**: 高。③ 的决策记录(session-report 决策⑤→③→①)显示用户确认全进,但问题在于"全进"本身是否应为 MVP。若架构阶段核算隐式求解器工程量(autodiff + 牛顿迭代 + LU 分解 + 活动集约束求解)超预期,将无缓冲空间。

**建议**:  
- 在 5.1 中为隐式求解器标注风险标识,而非仅藏于 addendum
- 明确"若架构核算超载,求解器降级为自适应 RK45 仍属 MVP"

### 1.4 🔶 量纲一致性校验 + 模型级时间单位
**严重度**: Minor  

**Challenge**: 量纲校验为软警告(不阻断运行),对教育场景有价值但非阻塞性。Addendum §8.3 将其列为第三优先降级杠杆(⑤>③>①)。若 MVP 体量必须瘦身,量纲从 C(完整推导)降为 B(显示用 units)是合理选择,且不影响核心建模+仿真体验。

**决策惯性嫌疑**: 中。用户确认方案 C 进 MVP,但软警告性质本身暗示其优先级应低于阻塞性功能。

**建议**: 维持,但应明确标注降级选项。

### 1.5 🔶 CRDT 实时多人协作 (含公式语义感知合并)
**严重度**: Major  

**Challenge**: 拆分两个层级:
- **多人协作** (CRDT 中继 + Awareness + 房主机制): MVP 必要。实时协作是核心差异化。
- **公式语义感知合并** (结构化 AST CRDT + AST↔文本双向同步 + 待定节点): addendum §8.3 自认"协作编辑器领域硬骨头",且列为第一优先降级杠杆(⑤>③>①)。10 人匿名教育场景下,并发编辑同一公式的概率极低;方案 A(检测+提示)对 MVP 已足够。

**决策惯性嫌疑**: 高。用户确认方案 C,但 10 人并发编辑同一流量公式的概率在统计上可忽略,为此付出协作编辑器领域最高复杂度,ROI 存疑。

**建议**: 在 5.1 中标注"语义感知合并为含项,但若架构核算超载,降级为冲突检测+提示(方案 A)仍维持 MVP 完整性"。

### 1.6 ✅ 房主计算 + 迁移机制 (从服务端仿真态快照续跑)
**判定**: MVP 必要。多人协作模拟的必要基础设施。零争议。

### 1.7 ❓ 跨画板复制粘贴 (带协议校验)
**严重度**: Observation  

**Challenge**: 对本 MVP 价值主张("轻量教育工具")为增强性功能,非核心。协议校验(CRC32 + 结构白名单)增加实现复杂度。粘贴失败不阻塞建模。

**建议**: 可降级为后续,或标注"若体量超载可降级,不削弱核心产品故事"。

### 1.8 ✅ 画板独立撤销栈
**判定**: MVP 必要。标准 UX 期望,对协作场景尤为重要。零争议。

### 1.9 ✅ 存在感知 (光标 + 选区跟随)
**判定**: MVP 必要。多人协作的可见性基础。零争议。

### 1.10 ✅ 内联 ASCII 趋势可视化 (sparkline)
**判定**: MVP 必要。纯字符渲染,实现轻量,是 MVP 层次(非 L3)的仿真结果呈现。零争议。

### 1.11 ✅ 可选游戏化层:行为徽章系统 + 总开关
**判定**: MVP 可接受。基于动作触发(不依赖仿真结果比对),实现轻量;有关闭开关。零争议。

## 3. 核查点三:体量警示逃生阀缺失

**严重度**: Critical (若架构阶段才发现超载,无 PRD 级授权回调)

**现状**:
- 四次体量警示均记录于 addendum (§7.3 求解器 + §8.3 语义合并 + 两处累计警示)
- 减负杠杆优先级已明确: ⑤>③>① (语义合并降级 > 求解器降级 > 量纲降级)
- 但 PRD 主文档 5.x 完全未提及逃生阀

**问题**: PRD 是跨角色(PM/架构/工程/设计)的权威参考文档。仅阅读 PRD 的人(典型场景:架构师做阶段评审、新成员加入)无法知晓:
1. MVP 范围存在超载风险
2. 有预先设定的减负优先级
3. 哪些项是"含但可降级"(语义合并⑤),哪些是"硬含"(画布、图元)

**建议**: 在 PRD 5.1 末尾或 5.2 之前插入显式逃生阀段落:

```
[体量逃生阀] MVP 四项体量警示已记录(addendum §7.3/§8.3)。
若架构阶段核算判定 MVP 范围超载,按以下优先级降级:
  ① ⑤ 公式语义合并 → 降为冲突检测+提示(方案 A)
  ② ③ 隐式求解器 → 降为自适应 RK45
  ③ ① 量纲校验 → 降为显示用 units(方案 B)
  ④ 跨画板复制粘贴 → 移至后续
降级后仍维持核心产品故事:ASCII SFD 建模 + 多人协作 + 数值模拟。
```

## 4. 核查点四:开放问题清单

### 开放问题 2: 官方支持的最大画板尺寸?
**判定**: 真开放。10,000 图元为当前目标,需架构阶段做性能 profile 后确认。无隐藏答案。

### 开放问题 3: MVP 仅匿名协作,还是需要用户认证系统?
**判定**: 真开放。此问题影响 FR-GAME-2 徽章持久化与画板归属策略。5.2 排除"用户账户与权限系统"暗示倾向匿名,但未收口。需在架构阶段前决策,因其影响存储模型。

### 开放问题 4: 挑战关卡达标判定引擎 vs L3 时序面板数据通道复用?
**判定**: 真开放。两者均为后续能力,不阻塞 MVP。Addendum §4 已正确标注"若 MVP sparkline 采样缓冲设计得当可为复用铺路"。**非 phase-blocker**——决策可推迟至挑战关卡或 L3 任一方开始实现时。

### [ASSUMPTION 1]: 模拟主机为第一个进入房间的用户
**判定**: 已记录合理假设。替代方案(服务端统一计算)已在 addendum 被拒。无问题。

### [ASSUMPTION 2]: 粘贴后不自动选中
**判定**: 已记录合理假设。建议保留实现阶段确认。

### [ASSUMPTION 3]: 粘贴原子性策略待定
**判定**: 真开放。选项 A/B 均合理,建议架构阶段定。

## 6. 核查点六:C3/C4 "后续"边界清晰度

### C3: CLD 后续
**判定**: 边界清晰,无回流风险。

- Addendum §5 明确 CLD 为独立图元系统(变量节点 + 因果链接 +/- 极性),不混入 SFD
- 已删除项明确标注"不回归": flow 的 polarity 字段、⊕/⊖ 反馈标识——即使 CLD 引入也是 CLD 专属图元
- MVP 的 FR-CANVAS-3/FR-ELEM-3 已无 CLD 残余

### C4: 挑战关卡后续
**判定**: 边界清晰,无回流风险。

- FR-GAME-3 明确"MVP 不含挑战关卡"
- 依赖链已识别: 挑战关卡 → 仿真结果数值比对引擎(后续前置依赖)
- Addendum §6.3 将关卡编辑器/社区分享标识为"更后续项",进一步防范围蔓延
- MVP 的 FR-GAME-2 徽章仅基于动作,与仿真结果比对无关,不存在"渐进式滑入"
